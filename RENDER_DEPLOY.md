# Deploying LGTM to Render

This repo ships with a [`render.yaml`](./render.yaml) Blueprint that provisions
the **backend**, **frontend**, and **Redis** in a single click. MongoDB is not
managed by Render — use MongoDB Atlas (free) and paste the connection URI
during setup.

## 0. Prerequisites — services you must create first

Render will **not** create these for you. Have them ready before you start.

| Service | Why | Where |
|---|---|---|
| **MongoDB Atlas cluster** | App database (Render does not host Mongo) | https://cloud.mongodb.com → free M0 cluster → Database Access (create user) → Network Access (allow `0.0.0.0/0`) → copy the `mongodb+srv://…` URI |
| **GitHub OAuth App** | User sign-in | https://github.com/settings/developers → New OAuth App. Callback URL is set in step 4 |
| **GitHub App** (separate from OAuth App) | PR review bot | https://github.com/settings/apps → New GitHub App. Webhook URL is set in step 4 |
| **SMTP credentials** *(optional)* | Email notifications | Any provider (SendGrid, SES, Gmail App Password). Set `EMAIL_DISABLED=true` to skip |
| **Sentry DSN** *(optional)* | Error tracking | https://sentry.io |
| **Dodo Payments keys** *(optional)* | Billing | https://dodopayments.com |

---

## 1. Push the repo to GitHub

Render reads `render.yaml` from a connected GitHub/GitLab repo.

```bash
git add render.yaml client/public/_redirects RENDER_DEPLOY.md
git commit -m "Add Render Blueprint"
git push origin main
```

---

## 2. Create the Blueprint on Render

1. Go to https://dashboard.render.com → **New** → **Blueprint**.
2. Connect the GitHub repo.
3. Render reads `render.yaml` and shows three resources:
   - `lgtm-redis` (Redis)
   - `lgtm-server` (Web Service — Docker)
   - `lgtm-client` (Static Site)
4. Click **Apply**. Redis comes up in ~1 min; the server build takes 5–10 min
   (tree-sitter native bindings compile during the Docker build).

**Tip:** the first server deploy will fail in the health check because env
vars aren't set yet. That's expected — finish step 3 below, then click
**Manual Deploy → Deploy latest commit**.

---

## 3. Set environment variables

Open each service in the Render dashboard → **Environment** tab.

### `lgtm-server` — required vars

| Key | Value |
|---|---|
| `MONGODB_URI` | `mongodb+srv://user:pass@cluster.mongodb.net/lgtm?retryWrites=true&w=majority` |
| `CLIENT_URL` | `https://lgtm-client.onrender.com` *(filled in after step 4)* |
| `GITHUB_APP_ID` | from your GitHub App settings |
| `GITHUB_APP_PRIVATE_KEY` | paste the full `.pem` contents, including `-----BEGIN…-----` and `-----END…-----` lines |
| `GITHUB_APP_SLUG` | the slug in your App's URL |
| `GITHUB_CLIENT_ID` | from your OAuth App |
| `GITHUB_CLIENT_SECRET` | from your OAuth App |
| `GITHUB_WEBHOOK_SECRET` | the secret you set on the GitHub App webhook |

`JWT_SECRET` and `JWT_REFRESH_SECRET` are auto-generated by the blueprint —
don't change them.

`REDIS_URL` is wired automatically from the `lgtm-redis` service.

Optional (but recommended):

| Key | Value |
|---|---|
| `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASS` | email server, or set `EMAIL_DISABLED=true` |
| `SENTRY_DSN` | error tracking |
| `DODO_PAYMENTS_API_KEY` / `DODO_PRODUCT_ID` / `DODO_WEBHOOK_SECRET` | billing |

### `lgtm-client` — required vars

| Key | Value |
|---|---|
| `VITE_API_URL` | `https://lgtm-server.onrender.com` *(no trailing slash)* |
| `VITE_SENTRY_DSN` *(optional)* | Sentry DSN |
| `VITE_DODO_ENV` *(optional)* | `live` or `test` |

After setting these, hit **Manual Deploy → Deploy latest commit** on each
service so they pick up the env vars.

---

## 4. Wire up the public URLs

Render assigns each service a URL like `https://<name>.onrender.com`. Once
both services are deployed:

1. Copy the frontend URL (`https://lgtm-client.onrender.com`) → paste it into
   `lgtm-server`'s `CLIENT_URL` env var. Redeploy the server (CORS reads this).
2. Copy the backend URL (`https://lgtm-server.onrender.com`) → paste it into
   `lgtm-client`'s `VITE_API_URL`. Redeploy the client (Vite bakes env vars
   into the build at build time).
3. Update **GitHub OAuth App** → Authorization callback URL:
   `https://lgtm-server.onrender.com/auth/github/callback`
4. Update **GitHub App** → Webhook URL:
   `https://lgtm-server.onrender.com/webhooks/github`
5. Update **Dodo Payments** webhook (if used):
   `https://lgtm-server.onrender.com/webhooks/dodo`

---

## 5. Verify the deploy

```bash
# health check
curl https://lgtm-server.onrender.com/healthz
# {"status":"ok","timestamp":"…","db":true}

# frontend loads
open https://lgtm-client.onrender.com
```

In the Render dashboard, server logs should show:

```
[Sentry] Initialized
[LGTM] Context worker initialized
[LGTM] Review worker initialized
[LGTM] Security worker initialized
[LGTM] Server running on 0.0.0.0:3000
```

If `Context/Review/Security worker NOT initialized` appears, `REDIS_URL`
isn't reaching the service — re-link the Redis instance under Environment.

## 6. Plan / cost notes

| Resource | Free tier behavior | Recommendation |
|---|---|---|
| **lgtm-server** | Free web services sleep after 15 min idle and have 512 MB RAM | The blueprint defaults to the **Starter** plan (`$7/mo`). Free tier sleeping breaks BullMQ workers — keep Starter |
| **lgtm-redis** | 25 MB free, no eviction, persists 30 days | Free tier is fine to start. Upgrade if queue depth grows |
| **lgtm-client** | Static sites are free, unlimited bandwidth on Render | No upgrade needed |
| **MongoDB Atlas M0** | 512 MB storage, shared CPU, free forever | Upgrade to M10 once you have real usage |

## 7. Common gotchas

- **`maxmemoryPolicy` must be `noeviction`** for Redis — already set in the
  blueprint. BullMQ refuses to run otherwise.
- **`GITHUB_APP_PRIVATE_KEY`** must include literal newlines. Render's env
  editor preserves them; just paste the whole `.pem` body.
- **Vite env vars are baked at build time.** Changing `VITE_API_URL` requires
  a **rebuild** (Manual Deploy), not just a restart.
- **Socket.io** works out of the box — Render supports WebSockets natively.
  No extra config needed.
- **Cold starts:** Render Starter plan keeps the server warm. If you ever
  drop to Free, expect 30–60s cold starts and dead BullMQ workers.
- **Health check failures** during first deploy are normal — env vars
  aren't set yet. Set them, then redeploy.
- **CORS errors in browser:** double-check `CLIENT_URL` on the server
  matches the frontend URL **exactly** (no trailing slash, https not http).

---

## 8. Rolling back / re-deploying

- Each service has a **Manual Deploy** dropdown — pick a previous commit to
  roll back.
- Pushes to `main` auto-deploy (we set `autoDeploy: true`). Disable per
  service if you want a manual gate.
- To wipe everything and redo: **dashboard → service → Settings → Delete**.
  Redis data is gone but Mongo persists.
